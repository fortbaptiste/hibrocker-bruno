Section 03 — Le constat

Le probleme aujourd'hui.

Ce que vivent les courtiers au quotidien, et pourquoi ca doit changer.

✗

Avis negatifs non maitrises

Un client mecontent va directement sur Google. L'avis negatif est visible par tous en quelques secondes. Le courtier le decouvre parfois des semaines plus tard, quand le mal est fait. Pas de filtre, pas de mediation, pas de chance de resoudre le probleme avant qu'il devienne public. La note moyenne chute, et avec elle, la confiance des prospects qui cherchent un courtier sur Google Maps.

✗

Aucune tracabilite ACPR/FSMA

Les courtiers en assurance ont l'obligation legale de tenir un registre des reclamations. En France c'est l'ACPR (recommandation 2024-R-02), en Belgique c'est la FSMA. Accuse de reception sous 10 jours, reponse sous 2 mois (1 mois en Belgique). Aujourd'hui, la plupart des courtiers gerent ca sur un fichier Excel, quand ils le font. En cas de controle, c'est la panique.

✗

Reseaux sociaux abandonnes

Les courtiers savent qu'ils devraient etre presents sur Facebook, LinkedIn, Instagram. Mais ils n'ont ni le temps, ni les competences, ni le contenu. Resultat : des pages vides ou des publications datant de 6 mois. La plateforme actuelle Hi Broker Connect repondait a ce besoin, mais souffre de deconnexions frequentes (surtout Meta) et de problemes de stabilite qui frustrent les utilisateurs.

✗

Deux outils separes

Aujourd'hui, la gestion des avis (Advice) et la multidiffusion reseaux sociaux (Connect) sont deux projets distincts. Ca veut dire deux onboardings, deux interfaces, deux logins. Pour un courtier qui a deja peu de temps, c'est un frein a l'adoption. L'ideal, c'est un seul endroit qui regroupe tout : reputation, avis, reclamations, reseaux sociaux, statistiques.

Section 06 — Dimensionnement

MVP : 5 courtiers.

On commence petit, on valide tout, et on scale ensuite. Voici pourquoi 5 courtiers est le bon chiffre pour demarrer.

01

Valider les integrations

5 courtiers suffisent pour tester l'OAuth Facebook, Instagram, LinkedIn et Google Business. On verifie que les connexions tiennent, que les publications passent, que les tokens se rafraichissent correctement. C'est le point critique identifie sur la plateforme actuelle : les deconnexions Meta. Avec 5 courtiers, on peut surveiller chaque connexion de pres et corriger les problemes avant de passer a 800.

02

Tester le parcours avis

Avec 5 courtiers reels, on aura de vrais clients qui passent par la page tampon, de vrais avis positifs et negatifs, de vraies reclamations a traiter. Ca permet de valider tout le flux : l'email de sollicitation, la page tampon, la logique de routage, le registre ACPR/FSMA, les alertes, l'analyse IA. Sur 5 courtiers, on peut estimer recevoir 20 a 50 avis par mois, ce qui est suffisant pour tester.

03

Rester dans les free tiers

5 courtiers, ca tient dans les offres gratuites de Supabase (500 MB), Brevo (300 emails/jour), et les quotas API. Les couts de fonctionnement restent minimaux pendant la phase de validation. Quand on passera a 50, 100, puis 800 courtiers, on upgradra les plans progressivement. Mais pour le MVP, on ne depense pas un euro en infrastructure inutile.

Section 07 — Architecture technique

Comment c'est construit.

Pas de Bubble. On code en dur avec des technologies modernes, open-source, et scalables. C'est mieux pour les automatisations, la flexibilite, les performances, et les integrations OAuth.

Pourquoi pas Bubble ? Le CDC original mentionne Bubble pour le front-end. Je recommande Next.js a la place pour plusieurs raisons concretes : les integrations OAuth (Facebook, Instagram, LinkedIn, Google) sont beaucoup plus fiables en code natif qu'a travers les connecteurs Bubble. La multidiffusion sur 4 reseaux sociaux necessite un controle fin des API calls et du rate limiting. Et surtout, pour gerer 800 courtiers a terme, les performances de Bubble montrent leurs limites (temps de chargement, cout par workflow).

Next.js donne aussi un controle total sur le SEO, le SSR (Server-Side Rendering), et les API routes. Ca veut dire que la page tampon client (celle ou on donne son avis) sera rapide, bien referencee, et personnalisable a l'infini. Et en cas de besoin, on peut toujours connecter une interface Bubble par-dessus via l'API REST de Supabase.

Section 08 — Supabase

La base de donnees.

S

Supabase

Supabase, c'est du PostgreSQL heberge dans le cloud avec plein de fonctionnalites deja integrees : authentification, API REST automatique, stockage de fichiers, et realtime. En gros, c'est une alternative open-source a Firebase, mais en mieux parce que c'est du vrai SQL derriere.

Pour Hi Broker, Supabase gere tout : les comptes courtiers, les avis, les reclamations, les publications, les connexions OAuth, les stats. L'API REST est generee automatiquement a partir du schema de base de donnees, ce qui accelere enormement le developpement.

Section 10 — Next.js

Le front-end.

Next.js est le framework React de reference. Il permet de construire des interfaces rapides, referencees, et scalables. C'est ce que utilisent Vercel, Netflix, TikTok, et des milliers de SaaS.

Ce que Next.js gere

  • ✓ Interface courtier : dashboard, avis, reclamations, reseaux, parametres
  • ✓ Interface admin : creation de contenu, gestion courtiers, monitoring
  • ✓ Page tampon client : formulaire d'avis, note, commentaire
  • ✓ API routes : webhooks Brio/Penbox, callbacks OAuth, endpoints internes
  • ✓ SSR : pages pre-rendues cote serveur pour la performance et le SEO
  • ✓ Responsive : fonctionne sur desktop, tablette et mobile

Pourquoi pas Bubble

✗
OAuth limites

Les connecteurs OAuth de Bubble sont generiques et fragiles. Pour gerer 800 connexions Meta/LinkedIn, il faut du code natif.

✗
Performance

Bubble charge lentement avec beaucoup de donnees. Un dashboard avec 800 courtiers sera penible.

✗
Cout

Bubble facture par workflow et par capacite. A l'echelle, ca coute plus cher qu'un VPS.

Section 11 — OpenAI

L'intelligence artificielle.

GPT-4o-mini analyse chaque avis, detecte le sentiment et les themes, et genere des propositions de reponses dans differents tons. Le tout pour quasi rien en cout.

Ce que l'IA fait

Analyse de sentiment
Chaque avis est classe : positif, neutre, negatif. Avec un score de -1 a +1.
Detection de themes
Delai, prix, communication, disponibilite, conseil, suivi, sinistre, accueil.
Points de friction
Identification precise de ce qui a deplu au client.
Generation de reponses
3 tons : professionnel, empathique, direct. Le courtier copie-colle celle qu'il prefere.

Section 13 — Scraping Google

La veille automatisee.

Chaque jour, la plateforme collecte les avis Google du courtier et de ses concurrents. Automatiquement. Sans intervention humaine.

Avis Google du courtier

01 Cron n8n tous les jours a 6h du matin
02 Appel API Apify avec le place_id Google du courtier
03 Recuperation de tous les avis (auteur, note, texte, date, reponse)
04 Comparaison avec les avis deja en base (detection nouveaux/modifies/supprimes)
05 Si nouvel avis ≤ 2 etoiles → alerte email immediate au courtier
06 Si avis positif lie a une sollicitation → liaison avis interne ↔ Google

Avis des concurrents

01 Detection automatique des courtiers dans la zone definie (ex: Bruxelles + 20km)
02 Scraping hebdomadaire de leurs avis Google
03 Suivi de leur note moyenne, volume d'avis, evolution mensuelle
04 Analyse IA des themes recurrents chez les concurrents
05 Alerte si un concurrent depasse la note du courtier
06 Tableau comparatif dans le dashboard
Service utilise : Apify (5€/mois de credit gratuit) ou SerpAPI (100 requetes/mois gratuites). Pour 5 courtiers + 15-20 concurrents, le free tier suffit largement.

Section 17 — Connexions OAuth

Les reseaux sociaux.

Le courtier connecte ses comptes Facebook, Instagram, LinkedIn et Google Business en un clic. C'est la partie la plus critique du projet — celle qui posait probleme sur la plateforme actuelle.



Facebook + Instagram

La connexion passe par l'API Meta Graph. Le courtier clique sur "Se connecter avec Facebook", autorise l'application, et on recupere un token d'acces qui permet de publier sur sa Page Facebook et son compte Instagram Business.

Permissions demandees :
pages_manage_posts — publier sur la Page
pages_read_engagement — lire les stats
instagram_basic — acces Instagram
instagram_content_publish — publier sur IG
Point critique : Les tokens Meta expirent apres 60 jours. La plateforme doit les rafraichir automatiquement AVANT expiration. C'est ce mecanisme qui manquait sur la plateforme actuelle et causait les deconnexions.
in

LinkedIn

LinkedIn offre deux types de publication : sur le profil personnel et sur la page entreprise. Le courtier choisit lesquels il veut activer. Le profil personnel necessite une validation explicite du courtier car ca publie en son nom.

Permissions demandees :
w_member_social — publier sur le profil perso
w_organization_social — publier sur la page entreprise
r_organization_social — lire les stats page
App review LinkedIn : L'application doit etre approuvee par LinkedIn pour acceder a ces permissions. Delai : 1-3 semaines. A lancer des le debut du projet.

Section 20 — Sollicitation

Demander un avis au client.

Apres chaque interaction (rendez-vous, sinistre, souscription), le client recoit un email l'invitant a donner son avis. Trois modes de declenchement possibles.

MODE A

Manuel

Le courtier ouvre Hi Broker, saisit l'email du client, selectionne le type d'interaction, et clique sur "Envoyer". L'email part immediatement via Brevo.

Quand l'utiliser : apres un appel telephonique, un RDV en face-a-face, ou quand le CRM n'est pas connecte.
MODE B

Campagne CSV

Le courtier importe un fichier CSV avec une liste de clients. La plateforme mappe les colonnes (email, prenom, nom, type interaction) et envoie les emails en masse.

Quand l'utiliser : pour solliciter une base existante de clients qui n'ont jamais ete contactes, ou apres un export mensuel du CRM.
MODE C

Webhook CRM

Le CRM (Brio ou via Penbox) envoie automatiquement un webhook a Hi Broker apres chaque interaction client. L'email est envoye sans aucune action du courtier. Zero effort.

C'est le mode ideal. Le courtier ne fait rien. Tout est automatique. Chaque client recoit une demande d'avis apres chaque interaction.

Section 24 — Avis negatifs

Note ≤ 3 : mediation interne.

Le client est mecontent. Pas de redirection Google. L'avis reste interne. Le courtier est alerte immediatement et peut traiter le probleme avant qu'il devienne public.

Ce qui se passe automatiquement

1
Pas de redirection Google
Le client voit un message empathique : "Merci pour votre retour. Votre avis a ete transmis a votre courtier qui vous recontactera."
2
Stockage dans Supabase
L'avis (note, commentaire, infos client) est enregistre dans la table "avis" avec source = "interne".
3
Analyse IA
OpenAI GPT-4o-mini analyse le sentiment, detecte le theme (delai, prix, communication...), et genere une reponse empathique suggeree.
4
Creation reclamation ACPR/FSMA
Une ligne est automatiquement creee dans le registre des reclamations avec tous les champs preremplis.
5
Alerte email au courtier
Le courtier recoit un email avec : la note, le commentaire, l'analyse IA, la reponse suggeree, et un lien vers la fiche reclamation.
Temps de reaction : Entre le moment ou le client soumet son avis et le moment ou le courtier recoit l'alerte email, il se passe moins de 30 secondes. C'est la force de l'automatisation n8n.

Section 30 — Multidiffusion

1 post → 20 publications.

L'admin planifie un post pour le 17 fevrier a 9h. A 9h, n8n prend le relais et publie automatiquement sur tous les reseaux de tous les courtiers eligibles.

Le workflow de publication

1
Cron n8n a l'heure planifiee
Recupere les publications prevues pour cette heure dans Supabase.
2
Filtre les courtiers eligibles
Si le post concerne Generali + IARD, seuls les courtiers qui ont coche les deux recoivent le post.
3
Genere les visuels personnalises
Pour chaque courtier, le visuel est adapte avec son logo et ses couleurs.
4
Verifie les connexions OAuth
Si le token Facebook de Cabinet Martin a expire → skip + alerte. Les autres courtiers ne sont pas impactes.
5
Publie sur chaque reseau
Appel API Facebook, Instagram, LinkedIn, Google Business. Avec gestion du rate limiting et des retries en cas d'erreur.
6
Stocke les resultats
Chaque publication est enregistree dans la table "diffusions" avec le post_id externe, le statut (succes/echec), et l'erreur eventuelle.
Pour le MVP (5 courtiers) : 1 post = 5 courtiers × 2-4 reseaux actifs = 10 a 20 appels API. C'est facilement gerable en quelques secondes. Pour 800 courtiers, on implementera une file d'attente pour etaler les publications sur quelques minutes et respecter les rate limits des APIs.

Pourquoi les deconnexions Meta arrivent

• Token expire : les tokens long-lived Meta durent 60 jours. Si on ne les rafraichit pas avant, perte de connexion.

• Mot de passe change : si le courtier change son mot de passe Facebook, tous les tokens sont revoques.

• Permissions revoquees : le courtier peut revoquer l'acces de l'app depuis les parametres Facebook.

• Meta revoque : Meta peut revoquer les permissions si l'app ne respecte pas les regles.

Notre solution : Refresh automatique des tokens a J-14, monitoring quotidien, alertes a J-7, parcours de reconnexion simplifie (1 bouton).

Section 48 — Points bloquants

Ce qu'il faut savoir.

Transparence totale. Voici les points qui peuvent bloquer ou ralentir le projet. Mieux vaut en parler maintenant que les decouvrir en cours de route.

BLOQUANT

App Review Meta (Facebook/Instagram)

Pour publier sur les pages des courtiers, notre app doit etre approuvee par Meta. Delai : 2 a 6 semaines. Action : soumettre l'app des le jour 1. En attendant, on peut tester avec des comptes de test.

RISQUE

Deconnexions OAuth Meta

C'est le probleme actuel de la plateforme. On peut le mitiger (refresh auto, monitoring, alertes) mais pas l'eliminer. C'est un probleme Meta, pas un probleme de code. L'objectif est de passer de "le courtier decouvre la deconnexion par hasard" a "le courtier est prevenu et se reconnecte en 1 clic".

RISQUE

Integration Brio (Portima)

Brio n'a pas de webhooks REST documentes publiquement. L'integration directe peut necessiter de passer par Penbox comme intermediaire, ou d'utiliser un fallback CSV/email. Action : contacter Portima/Brio pour obtenir la doc technique.

ATTENTION

App Review LinkedIn

LinkedIn exige une validation de l'app pour les permissions de publication. Delai : 1 a 3 semaines. Moins contraignant que Meta mais a anticiper.
